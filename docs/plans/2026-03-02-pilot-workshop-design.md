# Shodoboard Pilot - Workshop Design

**Date:** 2026-03-02
**Target:** PM training workshop, Wednesday 2026-03-04
**Author:** Jarkko Kailanto + Claude

## Context

Shodoboard is currently a scripted frontend prototype with hardcoded seed data (food delivery app scenario). For the Wednesday PM training workshop, we need it to work with real user data so participants can paste their own backlogs and get genuine value.

**Participants:** Finnish PMs, working individually on their laptops (possibly pairs).
**Language:** UI in English, user content in Finnish.

## What We're Building

A working PM coaching tool where workshop participants paste their own backlog, get AI-assisted outcome mapping, receive contextual coaching nudges, and export a structured board.

### Tech Stack
- **Frontend:** Next.js on Vercel (existing codebase)
- **Backend:** Next.js API routes
- **AI:** Anthropic Claude API (server-side, key in env vars)
- **Database:** Firebase/Firestore (one document per board)
- **Hosting:** Vercel (free tier)

## User Flow

### 1. Landing Page

- Hero with "Start" button (new real flow)
- "Try the demo" button (existing hardcoded prototype preserved)
- Keep existing branding and design

### 2. Consent Screen (Finnish)

Shown before intake. Explains in Finnish:
- What data is sent: backlog text and goals
- Where it goes: Anthropic's Claude API
- Anthropic's policy: not used for training, deleted within 30 days
- Board data stored in Firebase/Firestore
- Recommendation: don't paste personal data or trade secrets

Two buttons:
- "Ymmärrän ja jatkan" (I understand and continue) → AI intake
- "Jatka ilman tekoälyä" (Continue without AI) → manual form fallback

### 3. Intake Flow

**Input (two fields):**
1. Backlog (required): paste work items, tasks, tickets
2. Goals / OKRs (optional): existing business goals, OKRs, or outcomes

**AI Conversation (2-4 turns):**
1. AI identifies themes, suggests 2-3 business goals (or maps to provided goals). Asks for confirmation.
2. User adjusts.
3. AI proposes outcomes per goal, categorizes items as discovery/delivery, flags orphans.
4. User adjusts.
5. Board generated → saved to Firestore → redirect to `/board/[id]`.

**Design decisions:**
- 2-4 turns max (workshop time is limited)
- AI responds in English, understands Finnish content
- Proposals shown as structured/editable cards, not just chat text
- API route: `POST /api/intake`

### 4. Board (`/board/[id]`)

**Preserved from prototype:**
- 6-column kanban (Opportunities → Measuring)
- Goal → Outcome → Work Item hierarchy
- Drag-and-drop
- Dark/light mode
- Card detail modals

**New:**
- Dynamic URL with board ID (shareable, bookmarkable)
- Auto-save to Firestore on every state change
- "Saved" / "Saving..." indicator in header
- AI-generated nudges (~5) based on actual board data
- "Refresh nudges" button to re-analyze after changes
- Export button in header

### 5. AI Nudges

Generated by Claude after intake, based on actual board state. Examples:
- "This outcome has no measure of success yet"
- "All items under X are delivery - have you validated this?"
- "This goal has 8 items but no discovery work"

Nudges replace the current hardcoded ones. API route: `POST /api/nudge`.

### 6. Sparring (Nudge-Driven)

- Click "Spar on this" on any nudge → opens right sidebar
- AI acts as PM coach: asks questions, steers toward action
- Scoped to the specific nudge/item context
- **Hard limit: 4 user messages per session**
- After 2-3 exchanges, AI proposes concrete board changes
- "Apply to board" button updates the item directly
- Conversations NOT persisted (value is in the thinking)

**AI coaching persona knows:**
- Outcome-driven development
- Discovery vs delivery
- Measuring behavior change, not output
- OKR framing
- Assumption mapping

API route: `POST /api/spar`

### 7. Export

Structured markdown tree, downloaded as `.md` file:

```
# Board Export - [Board Name]

## Business Goals

### 1. [Goal statement]
- Timeframe: [...]
- Metrics: [...]

#### Outcomes
- **[Outcome statement]**
  - Behavior change: [...]
  - Measure of success: [...]
  - Work items:
    - [Discovery] Item title
    - [Delivery] Item title

## Items Not Yet Linked to an Outcome
- [orphaned items]

## Key Coaching Notes
- [Summary of active nudges]
```

Can also be printed to PDF via browser.

## Data Model

Existing `BoardState` type is preserved and stored as-is in Firestore:

```
Firestore collection: boards
Document ID: auto-generated (used in URL)
Document data: {
  createdAt: timestamp
  updatedAt: timestamp
  boardState: BoardState  // existing type from src/types/board.ts
  intakeHistory: ConversationMessage[]  // optional, for reference
  consentGiven: boolean
}
```

## Info Security Summary

| Concern | Answer |
|---------|--------|
| Where does data go? | Anthropic Claude API + Firebase/Firestore |
| Is data used for training? | No (Anthropic API policy) |
| How long is data retained? | Anthropic: up to 30 days. Firestore: until manually deleted. |
| Who can see a board? | Anyone with the URL (no auth) |
| API key exposure? | Key is server-side only (Vercel env var) |
| User consent? | Explicit Finnish consent screen before any AI use |
| Fallback without AI? | Manual form path available |

## What We're NOT Building

- User authentication / accounts
- Multi-user real-time collaboration
- Jira/Trello import integrations
- Board templates
- Admin dashboard
- Mobile optimization
- Sparring conversation persistence
